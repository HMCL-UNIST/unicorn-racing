> 자율주행 레이싱카에 쓰이는 **MPCC(Model Predictive Contouring Control)** 컨트롤러를 기본 개념부터 구현 구조까지 정리합니다. (개략적인 MPC/MPCC/MPPI 비교는 [MPC · MPCC · MPPI 비교]({{ site.baseurl }}/posts/mpc-mpcc-mppi/) 참고)

## 1. MPCC가 푸는 문제

레이싱카 제어의 목표는 한 문장으로 요약됩니다. **트랙을 벗어나지 않으면서 최대한 빨리 한 바퀴를 돈다.** 제어 문제로 옮기면 상충하는 두 욕심이 됩니다 — 빨리 가려고 하면 코너에서 라인 밖으로 밀려나고, 라인에 붙으려고 하면 느려집니다.

일반적인 MPC는 "시간표"를 따라갑니다. "0.5초 뒤엔 여기, 1.0초 뒤엔 저기"처럼 미리 정해둔 궤적을 시간 축으로 추종합니다. 레이싱에서는 이 시간표가 쉽게 깨집니다. 코너에서 조금 밀리거나 빨리 가면 예정된 시각의 예정된 위치와 어긋나고, 제어기는 시간을 맞추려 무리하게 됩니다.

**MPCC는 시간 대신 트랙 위 거리(호장, arc length $s$)를 기준으로 삼습니다.** "몇 초"가 아니라 "트랙 어디까지 왔나"로 생각합니다. 그러면 두 욕심을 그대로 최적화 문제로 적을 수 있습니다.

1. **Progress 최대화** — $s$를 최대한 빨리 늘린다 (빨리 달린다).
2. **Contouring error 최소화** — 라인에서 벗어난 양에 벌점을 준다.

핵심 오차는 두 가지입니다.

- **$e_c$ (contouring error):** 라인에서 **옆으로** 벗어난 거리.
- **$e_l$ (lag error):** 라인을 따라 **앞뒤로** 어긋난 정도.

MPCC는 매 제어주기마다 "지금부터 1~2초 미래까지 어떻게 달리면 가장 멀리 가면서도 라인에서 벗어나지 않는가"를 최적화로 풀고, 그 답의 **첫 한 스텝만** 실행합니다. 그리고 다음 주기에 다시 풉니다.

## 2. 시스템에서의 제어주기

```text
상태추정 ──── 차량 상태 (vx, vy, r, x, y, ψ) ──┐
글로벌 플래너 ── 레이싱 라인 ─────────────────┤
글로벌 플래너 ── 센터라인 (코리도어 경계) ──────┤
                                              ▼
                  ┌──────────────────────────┐
                  │  1. 현재 상태 x0 읽기       │
                  │  2. 가중치 갱신(곡률·장애물) │
                  │  3. acados로 OCP 풀기       │ ← 매 주기 도는 핵심
                  │  4. 미래 궤적에서 명령 추출   │
                  └──────────────────────────┘
                                              ▼
              주행 명령 (속도·조향) → 명령 먹스 → 모터 드라이버 → 모터
```

MPCC 노드는 **입력으로 차량 상태 + 레이싱 라인(추종 대상) + 센터라인(트랙 경계)**을 받고, **출력으로 속도·조향 명령**을 만들어 명령 먹스를 거쳐 모터로 보냅니다.

> **MPC는 매 주기 차량 상태를 받아 최적화 초기상태 $x_0$에 넣습니다.** 즉 위치·속도 피드백이 매 주기 들어갑니다. 이 사실은 8장에서 다시 중요해집니다.
{: .prompt-info }

## 3. 최적화 문제(OCP)를 어떻게 적는가

OCP(Optimal Control Problem) 정식화는 네 가지를 정의합니다: **상태**(무엇을 추적하나), **제어**(무엇을 조절하나), **동역학**(어떻게 움직이나), **비용·제약**(무엇이 좋고 무엇이 금지인가).

### 3.1 상태 — 8개

`x = [x, y, ψ, vx, vy, r, s, δ_prev]`

| idx | 심볼 | 의미 | 단위 |
|---|---|---|---|
| 0 | `x` | 전역 X 위치 | m |
| 1 | `y` | 전역 Y 위치 | m |
| 2 | `ψ` | 헤딩 | rad |
| 3 | `vx` | 차체 종방향 속도 | m/s |
| 4 | `vy` | 차체 횡방향 속도 | m/s |
| 5 | `r` | yaw rate | rad/s |
| 6 | `s` | progress(호장, 단조 증가) | m |
| 7 | `δ_prev` | 직전 적용 조향각 | rad |

`δ_prev`는 직전 조향각을 기억하는 칸으로, 지금 조향각과의 차이에 벌점을 줘 조향이 떨리지 않게 합니다.

### 3.2 제어 — 3개

`u = [a_x, δ, p_v]`

| idx | 심볼 | 의미 | 단위 |
|---|---|---|---|
| 0 | `a_x` | 종방향 **가속도** 명령 | m/s² |
| 1 | `δ` | **조향각** 명령 | rad |
| 2 | `p_v` | progress 속도(가상속도) | m/s |

중요한 점 — 제어변수는 속도가 아니라 **가속도** `a_x`입니다. 따라서 속도 명령은 솔버가 가속도를 적분해 만든 미래 속도 궤적에서 뽑아야 합니다(5장).

### 3.3 동역학: 저속 kinematic, 고속 dynamic, 그 사이는 혼합

저속에서는 타이어가 미끄러지지 않는다고 가정한 **kinematic 모델**, 고속에서는 미끄러짐을 반영한 **dynamic 모델**을 쓰고, 속도에 따라 부드럽게 섞습니다.

```python
w_dyn  = 0.5*(1 + tanh((vx - v_blend) / v_width))   # 속도↑일수록 dynamic 비중↑
f_expl = w_dyn*f_dyn + (1 - w_dyn)*f_kin
```

`tanh`는 S자 곡선이라 저속→고속 전환이 매끄럽습니다. 모델이 어느 속도에서 갑자기 튀면 솔버가 불안정해지므로, 부드러운 전환이 안정성에 직결됩니다. 저속에서 kinematic으로 떨어뜨리는 이유는 **dynamic 모델이 저속에서 수치적으로 불안정**하기 때문입니다(슬립각 `atan2(..., vx)`가 vx→0에서 발산).

**Kinematic 분기 (저속):**

```python
beta = atan(l_r * tan(δ) / L)   # 자전거 모델 슬립각
ẋ  = vx*cos(ψ + beta)
ẏ  = vx*sin(ψ + beta)
ψ̇  = (vx/L)*tan(δ)*cos(beta)
v̇x = a_x                         # 명령 가속도 = 가속 (지연 없음 가정)
ṡ  = p_v
```

**Dynamic 분기 (고속):**

```python
α_f = atan2(-vy - l_f*r, vx_safe) + δ   # 앞바퀴 슬립각
α_r = atan2(-vy + l_r*r, vx_safe)       # 뒷바퀴 슬립각
F_y = μ·D·F_z·tanh(B·α)                  # 타이어 횡력 (tanh로 포화)
```

슬립각이 커지면 횡력 $F_y$가 생기고, 한계를 넘으면 포화합니다(`tanh`로 모델링). `μ`는 노면 마찰계수입니다.

> `v̇x = a_x`는 "가속도를 명령하면 즉시 그대로 가속된다"고 가정합니다. 현실의 구동계 지연은 모델에 없습니다 — 8장에서 다시 다룹니다.
{: .prompt-warning }

### 3.4 비용함수: 가중치가 곧 제어기의 성격

여러 잔차에 가중치를 곱해 합산하고 $\tfrac{1}{2} r^\top W r$ 형태로 최소화합니다.

| 잔차 | 벌점 대상 |
|---|---|
| `e_c - e_c_ref` | contouring(가로 벗어남) |
| `e_l` | lag(세로 벗어남) |
| `yaw_err` | 헤딩 오차 |
| `vx - ref_v` | 속도 추종 |
| `δ - 곡률 피드포워드` | 조향 정규화 |
| `p_v - speed_target` | progress 보상 |
| `δ - δ_prev` | 조향레이트(평활) |
| `a_x` | 종가속 벌점 |

가중치는 절댓값보다 **상대 크기**가 중요합니다.

- **조향레이트 항을 크게** → 핸들 떨림 억제.
- **속도추종 > progress** → 보수적 주행 / **progress 키우면** → 공격적.
- **종가속 벌점 낮추면** → 공격적 가속.

**Terminal 비용**은 예측 구간 맨 끝에만 매겨 끝 지점 속도를 안착시킵니다.

### 3.5 Contouring / Lag 오차 계산

reference 라인은 호장 `s`로 파라미터화된 spline입니다. 오차 벡터를 라인 방향에 맞춰 두 성분으로 분해합니다.

```python
e_c =  sin_t*(x - ref_x) - cos_t*(y - ref_y)   # 라인에 수직 → 옆으로 벗어남
e_l = -cos_t*(x - ref_x) - sin_t*(y - ref_y)   # 라인을 따라 → 앞뒤로 어긋남
```

이렇게 분해하면 "옆 벗어남"과 "앞뒤 어긋남"에 다른 가중치를 줄 수 있습니다(보통 contouring을 더 강하게).

### 3.6 제약: 마찰타원이 물리 한계를 강제한다

**박스 제약** — `a_x`(최대 가·감속), `δ`(최대 조향), `p_v`(상한), vx ≥ 0(후진 금지).

**비선형 제약:**

1. **장애물** — 예측 궤적이 장애물을 침범하지 않게 비킨다.
2. **코리도어** — `e_c`를 좌/우 트랙 경계 사이로 가둔다.
3. **횡가속 한계** — `|a_lat| = |vx·r|`에 상한.
4. **마찰타원** —

$$
\left(\frac{a_x}{a_{lim}}\right)^2 + \left(\frac{a_{lat}}{a_{lim}}\right)^2 \le 1, \qquad a_{lim} = \mu \cdot g \cdot \eta
$$

   타이어 총 그립은 정해져 있어, 가속에 다 쓰면 선회에 못 씁니다. **코너에서 풀가속 + 풀선회는 물리적으로 불가능**하며, 이 타원이 그 한계를 강제해 솔버가 그립 배분을 자동 트레이드오프하게 합니다.
5. **조향레이트 한계** — hard 제약.

**스테이지별 속도 캡:** 각 지점 곡률 `κ`에 따라 `v_cap = sqrt(a_lat/|κ|)`로 상한을 걸어, 급코너 전부터 미리 감속합니다.

### 3.7 솔버 설정

| 항목 | 선택 | 의미 |
|---|---|---|
| 예측 지평 | 수십 스텝(약 1~2초) | 다음 코너 정도를 내다봄 |
| 적분기 | ERK (RK4) | 가속도 적분 → 속도·위치 |
| QP 솔버 | HPIPM | 매 반복 부분문제 풀이 |
| NLP 솔버 | SQP-RTI | 비선형 문제 풀이 |
| Hessian | Gauss-Newton | 2차 미분 근사 |

> **SQP-RTI(Real-Time Iteration)가 실시간성의 핵심입니다.** 보통 비선형 최적화는 수렴까지 여러 번 반복하지만, RTI는 매 주기 **한 번만** 반복해 "충분히 좋은 답"을 제때 냅니다. 부족분은 warm-start와 정규화로 보완합니다.
{: .prompt-tip }

## 4. 목표속도(ref_v)는 어떻게 만들어지나

글로벌 플래너가 발행하는 레이싱 라인(점마다 x, y, 곡률 κ, 목표속도 vx)을 받아 여러 상한을 거쳐 각 지점의 `ref_v`를 만듭니다.

```python
v_kappa = sqrt(a_lat_max / kappa)      # ① 곡률 기반 코너 속도 (급코너↓)
v_src   = wpnt.vx_mps * vel_scale       # ② 라인에 저장된 속도
ref_v   = clip(min(v_src, v_kappa, max_speed), v_min, max_speed)
# ③ 전진/후진 brake pass로 "감속만" 추가 보정
```

`a_lat_max`는 실제 타이어 그립 `μ·g·η`로 클램프됩니다. **μ를 실제보다 높게 잡으면** 모델이 그립을 과신해 코너 과속 진입 → 라인 이탈. 그래서 μ는 보수적으로 잡는 편이 안전합니다. **곡률 lookahead**로 먼 코너를 미리 반영해 도달 전부터 살짝 감속합니다.

## 5. 출력 단: 속도 명령을 어떻게 뽑나

제어변수는 가속도지만 모터 드라이버에는 **속도 명령**을 보내야 합니다. 솔버가 푼 **미래 속도 궤적에서** 뽑습니다.

**① accel-preview 창** — 약간 미래의 계획 속도를 명령

```python
v_brk = min(가까운 미래 구간의 계획 속도)
if v_brk < v_now:   v_plan = v_brk          # 감속 계획 → 즉시 감속
else:               v_plan = (앞선 계획 속도)  # 가속/유지 → 한발 앞선 속도
```

**② 제동거리-인지 출력 캡** — 솔버와 독립적인 안전장치

```python
v_allow = sqrt(vH² + 2·brake_anticip_a·d)   # d = 앞 코너까지 남은 거리
v_plan  = min(v_plan, min(v_allow))
```

**③ 최종 클램프** — 출발 보조 + 속도 상한, 조향은 EMA 필터·서보 슬루레이트 제한.

> 출력 속도를 사후에 임의 배율(×1.2 등)로 키우면 안 됩니다. 솔버는 "이 속도로 가다 저 지점에서 이만큼 감속"을 하나로 묶어 계획하므로, 배율을 곱하면 일관성이 깨져 코너에서 오버슈트합니다. 속도를 높이려면 **출력이 아니라 솔버 안에서** 조정해야 합니다.
{: .prompt-warning }

## 6. acados: OCP를 실시간으로 푸는 엔진

| 층 | 도구 | 역할 |
|---|---|---|
| 심볼릭 | **CasADi** | 동역학·비용·제약을 수식으로 적으면 미분 자동 생성 + C 코드 변환 |
| OCP | **acados** | 다단계 OCP 구조, 적분기, SQP/RTI, condensing. 전용 C 솔버 생성 |
| QP | **HPIPM** | 매 반복 QP를 푸는 고성능 내부점법 |
| 선형대수 | **BLASFEO** | 작은 dense 행렬 특화 라이브러리 |

빠른 이유: ① 코드 생성(전용 C 컴파일) ② condensing 구조 활용 ③ warm-start ④ BLASFEO. 정확하지만 느린 IPOPT도 선택지지만, 실시간 주행엔 acados를 씁니다.

## 7. 데이터로 보강하기: GP 잔차 보정 & BO 가중치 튜닝

### 7.1 GP 잔차 보정 — 모델 오차를 학습

```python
gp_z   = [vx, vy, r, δ, a_x]                          # GP 입력 특징
μ      = gp_posterior_mean(gp_z)                       # 학습된 모델 오차
f_expl = f_expl + [0, 0, 0, μ_vx, μ_vy, μ_r, 0, 0]    # 속도 상태에만 보정
```

핵심은 **속도 상태 `[vx, vy, r]`에만** 보정을 더한다는 점입니다. 오프라인에서 주행 데이터로 GP를 학습하고, 런타임에 사후 평균을 동역학에 더해 솔버에 통합합니다(학습 기반 레이싱 MPC의 표준 기법).

> GP는 모델의 *예측*을 정확하게 만듭니다. 다만 8장의 모델-액추에이터 불일치(차가 명령을 못 따라가는 경우)는 GP가 고치지 못합니다.
{: .prompt-info }

### 7.2 BO — 비용 가중치를 자동 튜닝

여러 가중치를 손으로 맞추는 건 비현실적이라 **BO(Bayesian Optimization)**로 탐색합니다: ① 가중치 조합 선택 → ② 한 바퀴 돌려 품질 점수(랩타임 − 벽위반·불안정 페널티) 측정 → ③ BO가 다음 유망 조합 예측 → ④ 반복 수렴. **적은 시도로 좋은 답**에 도달하는 게 핵심입니다.

> BO보다 더 중요한 건 **품질 점수 정의**입니다. 랩타임만 쓰면 벽을 긁어도 빠르기만 하면 좋게 학습하므로, 벽 위반·발산·정지에 페널티를 섞어 "빠르면서 안전한" 조합으로 유도합니다.
{: .prompt-tip }

## 8. 흔한 함정: 모델은 액추에이터가 빠르다고 가정한다

3.3의 `v̇x = a_x`는 "가속도를 명령하면 즉시 그만큼 가속된다"고 가정합니다. 현실에서는 명령이 모터 드라이버 내부 속도 루프를 거치는데, 이 안쪽 루프가 느리면 명령한 가속을 못 따라갑니다.

문제는 **상태 피드백이 이걸 고치지 못한다**는 점입니다. MPC는 매 주기 실제 속도를 $x_0$로 받아 "차가 지금 느리다"는 건 알지만, **"명령하면 그만큼 가속된다"는 모델 가정 자체는 피드백으로 갱신되지 않습니다.** 그래서 매 주기 또 낙관적으로 계획하고 또 못 따라갑니다.

이 가정에는 **위험한 비대칭**이 있습니다. 가속 쪽 오차는 비교적 무해하지만(출력은 제동 캡에 막힘), **제동 쪽 오차는 사고로 이어집니다** — 모델은 "코너 직전 강하게 밟으면 선다"고 믿는데 실제 제동이 약하면 코너를 벗어납니다.

| 방향 | 방법 | 결과 |
|---|---|---|
| A. 모델을 차에 맞춤 | 가·감속 한계를 실제 가능치로 낮춤 | 계획이 정직·제동 안전, 단 잠재성능↓ |
| B. 차를 모델에 맞춤 | 안쪽 루프를 빠르게(전류 직접 제어 등) | 솔버 계획 그대로 실현 → 성능 회복 |

제어이론으로는 안쪽 루프(속도 추종)가 충분히 빠르다는 **시간척도 분리** 가정이 깨진 상태입니다. 모델기반 제어를 실차에 올릴 때 가장 먼저 점검할 부분입니다.

## 9. 정리

- **시간이 아니라 트랙 거리($s$)를 기준**으로 삼는 것이 일반 MPC와의 결정적 차이.
- 제어변수가 **속도가 아니라 가속도**라, 속도 명령은 미래 궤적에서 뽑는다.
- **마찰타원**이 "코너에서 풀가속+풀선회 불가"라는 물리를 강제한다.
- **acados / HPIPM**으로 비선형 최적화를 실시간으로 푼다.
- 알고리즘이 완벽해도 **액추에이터가 명령을 못 따라가면 거기서 막힌다.**
