이전 글 [모션 모델(Motion Model)]({{ site.baseurl }}/posts/pf-motion-model/)에서는 오도메트리의 불확실성만큼 파티클을 맵 위에 퍼뜨려 보았습니다. 이제는 흩어진 수많은 파티클 중 '누가 진짜 내 위치에 있는지' 가려낼 차례입니다. 이때 사용하는 것이 바로 측정 모델(Measurement Model), 또는 센서 모델(Sensor Model)입니다.

모션 모델이 '내가 어디쯤 와 있을까?'를 추측하는 과정이라면, 측정 모델은 '내 눈앞에 보이는 이 풍경이 내 추측과 얼마나 일치하는가?'를 채점하는 과정입니다.

로봇은 라이다(LiDAR) 같은 센서를 통해 현재 주변 환경을 봅니다. 측정 모델은 각 파티클이 현재 자기 위치에 있다고 가정했을 때 관측될 가상의 스캔 데이터를 예상하고, 이를 실제 로봇이 관측한 데이터와 비교합니다.

두 데이터가 비슷할수록 해당 파티클이 실제 로봇의 위치일 확률이 높다고 판단하여 높은 가중치(Weight)를 부여합니다. 반대로 예상과 실제가 다르면 가중치를 깎습니다. 이 과정을 거쳐야만 이어지는 리샘플링 단계에서 진짜 위치에 가까운 파티클들만 살아남아 뭉치게 됩니다.

그렇다면 이 측정 모델을 어떻게 구현할까요?

## 2. 가상 라이다 매칭(Raycasting) 방식의 한계와 연산 병목

현재 우리가 구현한 가장 직관적인 센서 모델 방식은 각 파티클의 가상 자세에서 점유격자지도(Occupancy Grid Map)를 향해 가상의 레이를 쏘아 거리를 계산하는 레이캐스팅(Raycasting) 방식입니다.

우리가 가장 흔히 쓰는 레이캐스팅은 브레젠험(Bresenham)이나 DDA 알고리즘을 사용하여 격자 지도의 셀을 하나하나 전수조사하며 벽면을 찾아내야 하므로 연산량이 매우 무겁습니다.

- 파티클 개수: M개 (예: 1000개)

- 라이다 레이 개수: N개 (예: 채널당 360개)

- 총 연산량: 매 루프마다 O(M x N)번의 지오메트리 레이트레이싱 연산 필요

이 작업은 모션 모델 업데이트에 비해 비교가 안 될 정도로 무겁습니다. CPU 단일 스레드로는 실시간성(최소 20Hz 이상)을 확보하는 것이 불가능에 가깝습니다.

### 오린(Jetson Orin)에서 GPU 빌드가 강조되는 이유

다행히 레이캐스팅 방식은 각 파티클과 각 레이의 연산이 서로 완벽하게 독립적입니다. 즉, 당혹스러울 정도로 병렬화하기 쉬운(Embarrassingly Parallel) 구조를 가집니다. 수천 개의 코어를 가진 NVIDIA Jetson Orin 같은 임베디드 보드에서 CUDA 기반의 GPU 가속으로 빌드하면, 수천 개의 파티클에 대한 레이캐스팅을 동시에 처리할 수 있어 연산 속도를 획기적으로 끌어올릴 수 있습니다.

## 3. 연산을 줄이는 똑똑한 대안: AMCL의 룩업 테이블(Look-up Table) 방식

하지만 하드웨어 자원이 제한적이거나 GPU를 쓰지 않고 CPU만으로 빠른 연산 속도를 확보해야 한다면 어떻게 해야 할까요? ROS의 공식 표준 패키지인 AMCL(Adaptive Monte Carlo Localization)은 완전히 다른 패러다임인 가능도 필드(Likelihood Field / Distance Transform) 방식을 사용합니다.

### 핵심 아이디어: 가상 라이다를 쏘지 말자!

레이캐스팅이 무거운 이유는 '지도 내부를 통과하며 벽을 찾는 과정' 때문입니다. 그렇다면 반대로 지도의 모든 셀마다 가장 가까운 벽(장애물)까지의 거리를 미리 계산(Pre-computation)해 두면 어떨까요?

- 사전 연산 (Offline): 지도를 로드할 때, 거리 변환(Distance Transform) 알고리즘을 사용해 각 격자 셀에서 가장 가까운 벽까지의 거리가 적힌 룩업 테이블(Look-up Table)을 메모리에 구워둡니다.

- 실시간 연산 (Online): 센서 업데이트 루프가 돌면, 가상 라이다를 쏠 필요 없이 실제 차량이 쏜 라이다 스캔의 종점(End Point) 좌표를 각 파티클의 자세 기준으로 지도 위에 투영(Projection)만 합니다.

- 가중치 매칭: 투영된 종점 좌표의 격자 셀이 벽과 얼마나 떨어져 있는지 테이블에서 값을 O(1)로 즉시 읽어와 확률(가중치)을 계산합니다.

### 두 방식의 명확한 트레이드오프 (Trade-off)

| 비교 항목 | 레이캐스팅 (우리 방식) | 가능도 필드 (AMCL 방식) |
|---|---|---|
| 실시간 연산량 | 매우 높음 (격자 탐색) | 매우 낮음 (단순 좌표 투영 및 룩업) |
| 메모리(RAM) 점유 | 낮음 (기본 격자 지도만 소요) | 매우 높음 (정밀한 거리 필드 테이블 적재) |
| 하드웨어 최적화 | GPU 병렬 가속에 최적화 | CPU 멀티코어만으로도 충분히 실시간 구동 가능 |
| 동적 장애물 강건성 | 맵 뒤의 벽을 인지하므로 비교적 강인함 | 맵에 없는 장애물(상대차) 종점이 찍히면 오차가 커짐 |

## 4. 다양한 레이캐스팅 기법과 트레이드오프

위에서 언급한 가상 라이다 매칭도 단순히 격자를 하나씩 훑는 브레젠험(Bresenham) 방식만 있는 것은 아닙니다. 하드웨어와 메모리 제약에 따라 다음과 같은 다양한 최적화 기법을 적용할 수 있습니다.

- Bresenham / DDA: 방문하는 셀을 하나씩 검사합니다. 사전 계산이 필요 없고 맵 데이터만 있으면 되지만, 연산량이 많고 격자 단위의 정밀도를 가집니다.

- 레이 마칭 (Sphere Tracing): 거리 변환 맵을 1회 사전 계산해 둡니다. 각 위치에서 벽까지의 최소 거리만큼 빈 공간을 한 번에 크게 점프하며 탐색 속도를 높입니다.

- CDDT / PCDDT: 방향별 룩업 테이블을 생성해 압축 저장합니다. 2개를 조회하여 보간하는 방식을 쓰며, 가지치기를 통해 연산량을 줄입니다.

- Giant LUT: 모든 x, y, 각도(θ)에 대한 결괏값을 미리 다 구워버립니다. 1번의 O(1) 조회로 끝나 가장 빠르지만, 항목이 기하급수적으로 늘어나 엄청난 메모리(RAM)를 잡아먹습니다.

- 선분 해석 (Analytic): 격자를 쓰지 않고 선분 리스트를 직접 검사합니다. 격자 해상도에 무관하게 정확한 정밀도를 얻을 수 있습니다.

아래 데모에서 위 다섯 가지 레이캐스팅 기법을 직접 비교해볼 수 있습니다. 상단 탭으로 기법을 바꾸고, 캔버스에서 로봇을 드래그하거나 각도를 조절하면서 방문 셀 수·사전연산·메모리·정밀도의 트레이드오프가 어떻게 달라지는지 확인해보세요.

## 5. 가중치 계산 (Weight Computation)

가상의 라이다 스캔을 만들었거나 룩업 테이블에서 거리를 읽어왔다면, 이제 실제 라이다 스캔과 비교하여 점수를 매길 차례입니다.

이때 각 파티클의 가상 스캔과 실제 스캔의 차이를 가우시안 분포 식에 넣어 확률값을 구합니다.

$$
p(o_t \mid \bar{x}_t^i) = \frac{1}{\sigma\sqrt{2\pi}}\, e^{-\frac{(o_t - r^i)^2}{2\sigma^2}}
$$

여기서 $o_t$ 는 실제 관측값, $r^i$ 는 파티클 $i$ 가 자기 자세 $\bar{x}_t^i$ 에서 예상한 거리, $\sigma$ 는 센서 노이즈 표준편차입니다. 이 수식이 의미하는 바는 단순합니다. 파티클 위치에서 예상되는 스캔 데이터가 실제 관측 데이터와 가까울수록 해당 파티클은 기하급수적으로 더 큰 가중치를 받게 된다는 뜻입니다.

## 6. 리샘플링과 최종 위치 결정 (Resampling & Final Pose)

가중치 계산이 끝났다면, 점수가 높은 파티클을 살리고 낮은 파티클을 버리는 리샘플링을 수행합니다.

일반적으로 누적 가중치 분포 위에서 1/M 간격으로 균등하게 샘플링을 진행합니다. 이렇게 하면 가중치가 큰 파티클은 자연스럽게 여러 번 선택되어 다음 세대에 복제되고, 가중치가 0에 가까운 엉뚱한 파티클은 도태되어 사라집니다.

리샘플링을 거치면 살아남은 파티클들이 실제 로봇 위치 주변으로 조밀하게 군집을 이룹니다. 이때 리샘플링 후 가장 빈번하게 나타난 파티클의 위치를 선택하거나, 전체 파티클 분포의 기댓값을 계산하여 로봇의 최종 추정 포즈로 확정하게 됩니다.

## 7. 마무리

결국 측정 모델은 라이다 데이터를 바탕으로 수많은 가설들을 평가하고(가중치), 좋은 가설만 추려내어(리샘플링) 최종 정답을 도출하는 핵심 과정입니다.

자율주행 레이싱 스택을 설계할 때는 단순히 유명한 알고리즘을 가져다 쓰는 것을 넘어 하드웨어 아키텍처와 컴퓨팅 자원의 한계를 고려해야 합니다. 강력한 내장 GPU가 주어지는 환경이라면 병렬성이 극대화된 레이캐스팅 기반 스택을 고도화하는 것이 유리하며, CPU 연산 위주의 임베디드 환경이라면 메모리를 더 내어주더라도 O(1) 연산을 보장하는 거리 변환 기반의 AMCL 방식을 채택하는 것이 기술적으로 영리한 선택입니다.
